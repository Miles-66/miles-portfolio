# AI 写作评测工作台（LLM Eval Workbench）

一套面向**文学写作能力**的大模型评测系统，帮助编辑团队系统地审阅、归因、比较不同 LLM 的创作输出，并通过数据积累形成可迭代的编辑审美档案。

---

## 产品定位

本工作台的核心问题是：**哪个模型更会写作，差在哪里，为什么？**

传统的 LLM 评测框架（MMLU、HumanEval 等）无法回答这个问题。本系统围绕「编辑视角」设计，将人工审稿的判断结构化为可量化的控制点评分、失败归因和能力雷达，再通过 AI 辅助加速审稿流程，最终沉淀为可供模型后训练使用的高质量标注数据。

每一篇经过人工审阅的稿件，都携带控制点评分、批注片段、失败归因和能力缺口四个维度的标注信息，可直接作为偏好数据（preference data）或监督微调数据（SFT data）输出，为写作能力导向的模型训练提供稀缺的高质量语料。

当前数据规模：**15 个选题 · 129 篇稿件 · 10 个模型版本 · 2 个实验阶段**。

## 五大功能模块

### 1. 投稿箱（`/`）

编辑的日常工作入口。页面列出所有待审稿件，按选题分组展示，支持一键跳转到审稿页。右上角「归档」按钮可查看已完成选题的历史稿件，并按选题筛选。

### 2. 审稿页（`/eval/:id`）

系统的核心工作界面，采用左右分栏布局：

**左栏（稿件阅读区）**

- 支持「原文」与「批注」两种视图切换。批注视图通过 `AnnotatedText` 组件将已标注的文字片段高亮渲染（下划线、删除线、波浪线、陈词滥调四种类型）。
- 若模型在正文前输出了「预写阶段」内容（世界建构、破题意识等），在稿件上方折叠展示。
- **老编辑鉴定**：点击后调用 LLM 流式接口，对当前稿件的批注和评分草稿给出综合意见，并支持 `@老编辑` 追问——追问记录与编辑办公室的对话完全独立，按稿件 ID 持久化存储。老编辑在分析时会自动注入「编辑风格指南」（Layer 1），确保 AI 的判断口径与编辑本人的审美偏好保持一致。

**右栏（评分与归因区）**

- **控制点评分**：8 个维度，每项 1–5 分，附评语输入框。评分体系分 v1（基础层 40% + 核心层 60%）和 v2（情感与创意 38% + 语言与风格 40% + 工艺细节 12%）两套，系统根据选题的 `experimentPhase` 自动切换，无需手动选择。
- **自动归因**：点击「自动归因」后，LLM 读取稿件全文和当前评分，流式输出 `badcasePrimary`、`badcaseSecondary`、`failureLevel`、`capabilityGap` 四个字段的建议值，编辑可直接采纳或内联编辑。被采纳的自定义归因自动写入归因库，下次审稿时可复用。
- **参考正样本**：右侧面板展示与当前选题同类别（或同领域）的人工精选参考样本，供编辑对照判断。
- **训练数据标注**：可为每篇稿件标记训练数据类型和优先级，为后续模型训练提供数据基础。

### 3. 选题库（`/tasks`）

管理评测任务（选题）的全生命周期：

- **从 Google Sheets 批量导入**：粘贴 Sheets 链接后预览导入，系统自动检测冲突（已存在的 `taskId`），逐条选择覆盖或跳过。
- **手动新建选题**：填写任务类别、内容领域、难度、目标字数、目标风格、评测重点、Prompt 正文等字段。
- **委托写稿**：对任意选题，选择一个或多个模型（支持 OpenRouter 上的主流模型），设置 Prompt 条件（`none` / `full_scaffold` / `anti_cliche_only` 等），批量触发 LLM 写稿并实时查看进度。
- **AI 选题总评**：选题归档后自动触发，LLM 读取该选题下所有稿件，生成「共性问题」「最佳表现」「训练诊断」「改进方案」四项结构化报告，可手动编辑补充。

### 4. 编辑部简报（`/stats`）

基于全量审稿数据的可视化分析看板，解锁条件为审完一定数量的稿件。主要模块如下：

| 模块 | 说明 |
|---|---|
| **各模型审稿均分** | 按模型分组，展示加权综合均分和各维度均分，支持按 `promptCondition` 筛选 |
| **各维度得分分布** | v1/v2 两套维度分别展示，每个维度的 1–5 分分布直方图 |
| **模型能力雷达** | 选择两个模型进行两两对比，灰色虚线为全局均值基准；支持按实验阶段切换（`baseline` 使用 v1 维度，`v2-prompt-exp` 使用 v2 维度）；悬停维度轴标签时弹出 tooltip，显示该维度下分差最大的稿件对 |
| **退稿高频原因** | Top N 退稿原因及出现次数 |
| **退稿层级分布** | 按 `task_failure` / `paradigm_failure` / `control_failure` / `none` 分布 |
| **文体得分矩阵** | 按写作能力类别 × 模型的均分热力图 |
| **Cliché 之王排行榜** | 统计各模型被标注为「陈词滥调」的片段数量 |
| **选题意象同质化分析** | 选择一个选题后，LLM 提取所有稿件的比喻与意象，聚类找出跨模型共用的意象族 |
| **向量相似度热力图** | 计算同一选题下各模型输出的语义相似度，可视化为热力图 |

### 5. 编委办公室（`/office`）

编辑的「自我认知」与「AI 协作」空间，也是系统长期价值的核心沉淀区。包含两个并排区域：

**编辑档案（三层）**

| 层级 | 名称 | 说明 |
|---|---|---|
| Layer 1 | 编辑风格指南 | 评分模式、红线写法、正负面偏好；每次老编辑分析稿件时自动注入，确保 AI 判断口径与编辑审美一致 |
| Layer 1.5 | 模型能力档案 | 每个模型的固定印象和能力边界；做选题分析时按需注入，辅助横向比较 |
| Layer 2 | 审美人格 | 由 AI 读取全量审稿数据（评分、批注、归因、追问记录）后提炼的第三人称审美画像，包含矛盾、偏执和核心判断标准 |

每层档案显示生成时间、基于的数据量，以及距上次生成后新增的待处理稿件数，支持随时更新。随着审稿数据积累，档案的准确度和颗粒度持续提升。

**审美人格的长期价值**：Layer 2 审美人格画像是系统的核心资产之一。它不仅用于编辑自我认知，更是未来**自动化评分模型**的训练目标——通过将编辑的审美判断结构化、可重复地表达出来，为训练一个「懂这位编辑」的评分 agent 提供数据基础。当审稿数量足够多时，该画像可作为 system prompt 直接驱动自动化初审流程。

**老编辑对话**

持久化的 AI 对话窗口，历史消息跨会话保存。老编辑在对话中可访问编辑档案全部三层内容，支持深度探讨写作判断、归因逻辑、模型能力边界等议题。对话上下文与审稿页的追问记录相互独立，适合进行跨稿件的综合性分析。

### 附：样板间（`/references`）

管理人工精选的参考正样本库。每条样本包含标题、写作能力类别、内容领域、正文、来源和风格标签。审稿时系统按「优先匹配类别，次之匹配领域」的规则自动推荐 1–2 条参考样本。

## 评分体系

### v1（baseline 阶段，共 8 维）

| 层级 | 维度 | 权重 |
|---|---|---|
| 基础层（40%） | 任务完成度 | 15% |
| | 叙事连贯性 | 15% |
| | 风格一致性 | 10% |
| 核心层（60%） | 情感真实性 | 20% |
| | 可记忆性 | 15% |
| | 语言质感 | 12% |
| | 细节质量 | 8% |
| | 意象质量 | 5% |

### v2（v2-prompt-exp 及后续阶段，共 8 维）

| 层级（占比） | 维度 | 权重 | 说明 |
|---|---|---|---|
| **情感与创意**（53%） | 情感分寸感 | 20% | 克制与爆发的节奏是否得当，而非堆砌或过度渲染 |
| | 写作创意性 | 18% | 切入角度、结构或语言策略是否跳出惯常写法 |
| | 内核密度 | 15% | 核心意图是否清晰且有密度，每段是否推进核心 |
| **语言与风格**（35%） | 语言质感 | 15% | 用词精准度、句式变化、节奏感 |
| | 风格独特性 | 10% | 是否有辨识度，能否与其他模型输出明显区分 |
| | 物理真实性 | 10% | 身体感知、物理细节、空间描写是否真实可信 |
| **工艺细节**（12%） | 细节质量 | 7% | 细节是否充分且有说服力，而非泛泛而谈 |
| | 意象使用能力 | 5% | 意象是否具体鲜活且用得恰当，不流于装饰 |

综合得分由各控制点加权平均后，叠加编辑手动调整值（`overallAdjustment`）得出最终分（`finalScore`）。权重定义在 `shared/weights.ts`，前后端共用，启动时自动校验总和是否为 1.0。

## 失败归因体系

每篇稿件必须填写以下四个归因字段：

| 字段 | 说明 | 示例 |
|---|---|---|
| `badcasePrimary` | 主要退稿原因（10–20 字） | 平庸无亮点、语言陈腐 |
| `badcaseSecondary` | 次要退稿原因（可选） | 风格趋同/无声音 |
| `failureLevel` | 失败层级 | `task_failure`（任务理解失败）/ `paradigm_failure`（整体范型失败）/ `control_failure`（控制点失效）/ `none` |
| `capabilityGap` | 能力缺口（可选） | 情感节制能力不足 |

归因库（`custom_attributions` 表）自动收录被编辑采纳的自定义归因，按使用频次排序，下次审稿时可一键复用。

## 技术架构

```
前端          React 19 + Tailwind CSS 4 + shadcn/ui
后端          Express 4 + tRPC 11（类型安全 RPC）
数据库        MySQL（通过 Drizzle ORM 管理 Schema 和迁移）
AI 接入       invokeLLM()（服务端统一调用，不暴露密钥）
文件存储      S3（通过 storagePut/storageGet 封装）
认证          Manus OAuth（protectedProcedure 保护需登录接口）
```

### 数据库表一览

| 表名 | 说明 |
|---|---|
| `users` | 用户账号（Manus OAuth） |
| `tasks` | 选题（含 `experimentPhase`、`promptVersion`、`difficultyLevel` 等） |
| `eval_records` | 评测记录核心表（v1/v2 控制点评分、归因、训练数据标注，共约 40 个字段） |
| `annotations` | 稿件文本批注（字符偏移量定位） |
| `annotation_qas` | 审稿页 @老编辑 追问记录 |
| `task_reviews` | 选题 AI 总评（共性问题、最佳表现、训练诊断、改进方案） |
| `imagery_reports` | 意象同质化分析缓存 |
| `reference_samples` | 参考正样本库 |
| `editor_profiles` | 编辑档案（三层，Markdown 格式） |
| `editor_profile_history` | 编辑档案历史版本 |
| `editor_chat_messages` | 编委办公室持久化对话记录 |
| `custom_attributions` | 自定义归因库（使用频次统计） |

### 后端路由结构

后端路由按功能拆分到 `server/routers/` 目录下，主要模块包括：`tasks`（选题管理）、`evalRecords`（审稿记录）、`annotations`（批注）、`annotationQas`（追问）、`stats`（统计分析）、`commission`（委托写稿）、`diagnosis`（自动归因）、`taskReview`（选题总评）、`imageryCluster`（意象聚类）、`embeddingSimilarity`（向量相似度）、`referenceSamples`（参考样本）、`editorProfile`（编辑档案）、`editorChat`（办公室对话）、`customAttributions`（归因库）。

## 本地开发

```bash
# 安装依赖
pnpm install

# 启动开发服务器（前后端同端口，Vite 代理）
pnpm dev

# 数据库 Schema 变更流程
# 1. 修改 drizzle/schema.ts
# 2. 生成迁移 SQL
pnpm drizzle-kit generate
# 3. 在 webdev_execute_sql 工具中执行生成的 .sql 文件

# 运行测试
pnpm test
```

## 实验阶段说明

选题通过 `experimentPhase` 字段区分实验阶段，当前有两个阶段：

- **`baseline`**（12 个选题，119 篇稿件）：基准实验，无额外 Prompt 干预，使用 v1 评分体系。
- **`v2-prompt-exp`**（3 个选题，10 篇稿件）：Prompt 工程实验阶段，使用 v2 评分体系。

编辑部简报中的「模型能力雷达」支持按阶段切换，系统自动识别当前阶段使用的评分版本（v1 或 v2），无需手动配置。
